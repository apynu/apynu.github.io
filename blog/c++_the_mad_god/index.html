C++ - The mad God

I've been writing C++ for about 2500h now, which, by its standards, isn't a long time, but its a lot longer than most people who criticise it have.
In this post I want to talk about the good, the bad and the mad of C++ from the eyes of someone a little experienced.

1. The Bad

When giving critical feedback we should always pack it between two good things, though we're talking about C++ here, so we'll start with the bad continue with the good and end with the mad

1.1 Complexity

Its astounding how complex a language can get, and C++ has to be one of the most complex languages that I've ever used (and I prize myself with having used a bunch of different languages). The amount of features that it has makes it so one team could write their code with a completely different feature set than another team, which makes colaboration between teams difficult.

Not to mention that on can never know all features or C++

1.2 Technical Language Debt

-lots of old features that shouldn't be used: e.g. trailing return types

-makes it mandatory to maintain legacy code

1.3 Programmer speed

-due to how C++ works, programming in rust get results faster than in C++
